## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| **プロジェクト名** | auto-grants-integrated |
| **目的** | NPO/非営利団体向けの助成金・補助金の発見→マッチング→申請書作成→提出管理を一気通貫で支援するAI駆動プラットフォーム |
| **主要価値** | auto-grantsv2の全国NPO助成金管理基盤 + Subsidy Radarの自治体補助金インテリジェンス機能 + JobFlowのModal GPU AIインフラ + moneyflow-visualizerの予算フロー可視化を統合 |

### 1.1 統合の背景

本プロジェクトは4つの既存プロジェクトの強みを融合して構築する。

| プロジェクト | 強み | 課題 |
|---|---|---|
| **auto-grantsv2** | 全国NPO向け67 MCPツール、助成金検索・マッチング・申請書テンプレ管理が充実 | Embeddingは軽量モデル (Qwen3-0.6B / 384dim) でローカル実行のみ。セマンティック精度に限界 |
| **Subsidy Radar** | 富山県・市特化の自治体補助金収集（RSS/ページ差分/LLM抽出/交付金カスケード追跡）が強み | SQLiteでベクトル検索未対応。スケーラビリティに課題 |
| **JobFlow** | Modal上のQwen3-Embedding-8B (4096dim) とBgeRerankerが実証済み。サーバーレスGPUの相乗りでコスト追加なし | 助成金ドメインとは無関係 |
| **moneyflow-visualizer** | 国・自治体の予算フローをサンキーダイアグラム・3D地球儀で可視化。財務省・厚労省・こども家庭庁等10種の公式データソースを収集・DB化済み。Trusted Source Flow による信頼ソース選定フロー | ローカルSQLite + ローカルLLM (qwen2.5) によるRAGは精度に課題。バックエンドをauto-grants側のインフラに移行する必要あり |

### 1.2 統合目的

4つの強みを融合し、**最小コストで最大精度のセマンティック検索・マッチング**を実現する。さらに、国の予算編成から個別の助成金公募までの資金の流れを一気通貫で可視化し、**「なぜこの助成金が存在するのか」を予算の源流から理解した上で申請できる**トレーサビリティを提供する。

## 4. 非機能要件

### 4.1 パフォーマンス・応答性

| 項目 | 要件 |
|---|---|
| セマンティック検索レスポンス | 10秒以内（Modalウォームスタート時） |
| コールドスタート時 | 最大3分（Modal GPU起動含む）。この間、UIでは「AI起動中」のプログレス表示を行い、フォールバックとして部分一致キーワード検索の結果を暫定表示する。 |
| コレクター実行 | 各バッチ15分以内。ターゲットサイトの過負荷を避けるため、深夜時間帯（02:00〜05:00）に分散実行する。 |
| 同時接続 | 1-3ユーザー（個人/小規模NPO向け） |

### 4.2 可用性・開発環境

| 項目 | 要件 |
|---|---|
| 実行環境 | ローカル開発環境で完結（Supabase + Modal）。開発環境でも本番と同一のModal API（4096次元）を使用し、不要なローカル推論エンジンの構築を排除する。 |
| コスト最適化 | 自動スリープ/ウェイク機構。未使用時のModal起動コストは0円。 |
| Modal | min_containers=0でコールドスタート対応。 |

### 4.3 セキュリティ

| 項目 | 要件 |
|---|---|
| API認証 | API Key認証（Modal API / MCP API） |
| 機密管理 | 環境変数による機密情報管理（.env） |
| ユーザー認証 | Supabase Auth / Entra ID 対応 |

### 4.4 コスト

| 項目 | 要件 |
|---|---|
| Supabase | Free tier（個人開発規模） |
| Modal | JobFlowと共有（実質追加コスト0円） |
| 月間運用コスト目標 | **3,000円以内**（OPS_BUDGET_LIMIT_JPY） |

### 4.5 拡張性

| 項目 | 要件 |
|---|---|
| 地域拡張 | 他の自治体（石川県、沖縄県等）への地域拡張が容易 |
| Embeddingプロバイダ切替 | 環境変数1つ（EMBEDDING_PROVIDER）で modal/tei/ollama を切替可能。ただし、DBカラムの次元不整合を防ぐため、開発環境・本番環境ともに **原則 modal (4096dim) の使用を前提** とする。 |

## 5. 制約事項

| カテゴリ | 制約 |
|---|---|
| スクレイピング | 対象サイトの robots.txt を遵守。深夜帯に実行。 |
| リクエスト間隔 | 最低1秒以上 |
| 個人情報 | 収集・保存しない |
| 出典管理 | 出典URLと取得日時を必ず保持 |
| 規約制約 | ミラサポplus等の規約制約のあるサイトは月次の最小限チェックに留める |
