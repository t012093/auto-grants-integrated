## 1. プロジェクト概要

| 項目 | 内容 |
|---|---|
| **プロジェクト名** | auto-grants-integrated |
| **目的** | NPO/非営利団体および地域社会のための、資金調達（助成金・クラファン）から市民合意形成、プロジェクト実行・ボランティア管理までを一気通貫でサポートする市民参加型官民連携プラットフォーム |
| **主要価値** | 助成金RAG ＋ 予算トレーサビリティ ＋ 自治体プロポーザル（GraphRAG） ＋ 市民合意（Plurality） ＋ プロジェクト実行（ボランティア＆クラファン）の完全統合による、市民発プロジェクトの資金と行動の最大化 |

### 1.1 統合の背景

本プロジェクトは、以下の7つの既存プロジェクトおよび構想の強みを融合して、官民連携と市民協働の新しいインフラとして構築する。

| プロジェクト | 強み | 本プラットフォームにおける役割・統合価値 |
|---|---|---|
| **auto-grantsv2** | 全国NPO向け67 MCPツール、助成金検索・マッチング・テンプレート管理 | 助成金の基本検索・マッチング、および申請書生成エンジンとして継承。 |
| **Subsidy Radar** | 富山県・市特化の自治体補助金収集（RSS/ページ差分/LLM抽出） | 自治体レベルのローカルな公募情報を自動で収集・蓄積するインテリジェンス。 |
| **JobFlow** | Modal上のQwen3-Embedding-8B (4096dim) とBgeReranker | 開発・本番共通のサーバーレスGPU AIインフラとして、極めて低コストかつ高性能なベクトル処理を提供。 |
| **moneyflow-visualizer** | 国・自治体の予算フローの可視化、10種の公式予算データソース | 予算から助成金までの資金の流れを可視化し、なぜその助成金が存在するかの「予算源流」を示すトレーサビリティ。 |
| **govdoc-app** (GovPro Agent) | 自治体総合計画・予算書のGraphRAG構造化、BtoG提案書骨子自動生成 | 自治体の総合計画等に基づく「政策適合性」の分析、およびハルシネーションのないエビデンス（出典）付き提案書作成。 |
| **plurality-connect** (Plurality Pro) | デジタル民主主義（オードリー・タン理念）、市民合意形成、Quadratic Voting | 市民による合意形成と投票（協議スペース）。合意形成率を提案書のエビデンスとして自動連携。 |
| **volunteer-connect** | NPOオンボーディング、ボランティア案件応募・承認、クラウドファンディング | 実プロジェクトの実行管理、ボランティア・プロボノのマッチング、市民や企業からの直接的な資金調達（クラファン）。 |

## 2. ステークホルダー

| ロール | 対象 | 関心事 |
|---|---|---|
| **Primary User** | NPO法人 Open Coral Network（開発者兼利用者）などのNPO | 助成金の自動発見、申請書作成の効率化、ボランティア募集、市民からのクラファン資金調達。 |
| **Secondary User** | 一般市民・ボランティア・地域住民 | 地域のプロジェクトへの参加、意思決定（投票・合意形成）への参画、資金的支援。 |
| **Enterprise / Partner** | 企業の社会貢献（CSR）担当者、パートナー団体 | 地域プロジェクトへのプロボノ参加、企業の資金調達支援・協働。 |
| **Government** | 自治体職員・担当者 | 総合計画や施策に合致した優良なプロポーザルの獲得、市民合意が得られたプロジェクトの実施。 |
| **AI Assistant** | Claude Desktop / Code 経由 of AI | MCPツールやAPIを通じた、システムの代理操作と分析支援。 |

## 4. 信頼・プライバシー要件 (DID/ZKP)

本プラットフォームの全レイヤーにおいて、市民のプライバシー保護とデータの信頼性を担保するため、以下の非機能要件を定義します。

| ID | 要件名 | 説明 | 優先度 |
|---|---|---|---|
| NFR-901 | 自己主権型アイデンティティ (DID) | ユーザー（市民、NPO、自治体）はW3C標準のDIDを個別に保有し、プラットフォーム依存なしに自身の資格（Credentials）を署名・管理できること。 | P1 |
| NFR-902 | 属性のゼロ知識証明 (ZKP) | 住民資格、累計活動時間、保有資格の証明は、生データを開示せず、zk-SNARKs等のZKPを用いて「条件を満たしている」という事実のみを証明・検証可能にすること。 | P1 |
| NFR-903 | 行政文書署名検証 (DID) | 提案書ジェネレーター（GovPro）が参照する行政計画書やRFPは、自治体公式DID署名により改ざんがないことを機械的に検証できること。 | P1 |
| NFR-904 | 匿名合意形成 | 住民投票や協議コメント時、ユーザーの個人特定情報（PII）をマスクし、住民であるZKP証明のみで発言・投票の適格性を検証できること。 | P1 |

## 4. 非機能要件

### 4.1 パフォーマンス・応答性
* **セマンティック検索**: Modalウォームスタート時 10秒以内。コールドスタート時、最大3分（その間、進捗表示と暫定の全文検索結果を表示）。
* **UIアニメーション**: レンダリング速度を維持し、`backdrop-filter` や重いSVG描画が並行しても 60fps を維持する。
* **リアルタイム性**: 議論や投票結果、ボランティアの応募状況は Supabase Realtime により即座に反映される。

### 4.2 セキュリティ・プライバシー
* **ユーザー認証**: Supabase Authによる安全なサインイン（email/password）。
* **データ保護 (RLS)**: PostgreSQLの行セキュリティ (Row Level Security) により、NPOや企業パートナーの機密情報（非公開の申請書下書き等）への不正アクセスを徹底防御。
* **機密情報の秘匿**: 外部APIキー（OpenAI, Gemini, Supabase）は環境変数（`.env`）で管理し、絶対にデータベースやコードベースに直書きしない。

### 4.3 コスト
* **運用費目標**: 月間 3,000円以内（Free Tier の積極活用と Modal GPU の自動スピンダウン機構による）。

## 5. 制約事項
* **スクレイピングマナー**: robots.txt を遵守し、リクエスト間隔を最低1秒空けて深夜バッチにて実行。
* **依存ライブラリ**: フロントエンドは React 19 / Vite。グラフ描画は `@nivo/sankey`, `recharts`。3D地球儀は `react-globe.gl` を使用。
